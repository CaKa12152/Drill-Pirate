# Release and Support

This page is for maintainers distributing public alpha builds and collecting useful feedback.

## Release Versioning

Current app version:

```text
Alpha v2.7.0
```

Git tags should use the `v` prefix:

```text
v2.7.0
```

The in-app updater compares GitHub release tags against the app's current version.

## Public Release Checklist

Before publishing a release:

1. Update the app version in source.
2. Run the app from source once.
3. Create a test project.
4. Open an existing project.
5. Test audio playback.
6. Test Settings > Preferences and Settings > Devices.
7. Test project save/return home.
8. Test at least one formation tool.
9. Test one PDF export.
10. Test Project ZIP export.
11. Test MP4 export with a known ffmpeg path.
12. Run the Windows build script.
13. Launch the built EXE from `dist\Drill Pirate`.
14. Confirm the splash screen version.
15. Confirm plugin folder scanning works.
16. Zip and upload the Windows build to GitHub Releases.
17. Optionally upload a matching `.sha256` checksum asset for the Windows ZIP.
18. Put real release notes in the GitHub release description.

The app uses the GitHub latest release description for the post-update release log.

## Build Command

```powershell
.\build_exe.ps1
```

Expected build artifact:

```text
dist\Drill Pirate Alpha v2.7.0 Windows.zip
```

## What to Upload

Upload the Windows ZIP generated by the build script. Testers should extract the whole ZIP and run the EXE inside the extracted folder.

Do not upload only the raw `.exe` unless you have a separate installer that includes dependencies.

## Support Priorities

For alpha users, prioritize issues in this order:

1. Data loss or project corruption.
2. App launch failures.
3. Save/load failures.
4. Audio/playback breakage.
5. Export failures.
6. Broken core editing tools.
7. Plugin crashes caused by app API changes.
8. UI crowding or workflow polish.

## Privacy and Project Files

Drill Pirate stores projects locally. Project ZIPs may include:

- Audio files.
- Prop images.
- Performer names/labels.
- Drill coordinates.
- Plugin-generated data if plugins modify project files.

Tell users not to upload project ZIPs publicly if the show/music is private.

## Update Flow

The app checks:

```text
https://api.github.com/repos/CaKa12152/Drill-Pirate/releases/latest
```

Buttons:

| Button | Behavior |
| --- | --- |
| Install | Downloads and installs/launches the latest asset. |
| Skip This Version | Suppresses the current release tag. |
| Ignore | Dismisses the prompt for now. |

If the installed version is newer than the last recorded running version, Drill Pirate shows the release-log popup on first launch.

The update system supports:

- Stable and beta channels.
- Download size checks.
- Optional SHA-256 verification from checksum assets.
- ZIP validation before install.
- Rollback if the ZIP copy step fails.
- A local pending-release-notes marker so release logs survive the app restart.

## Known Distribution Gaps

- No signed installer yet.
- No auto rollback if an update fails midway.
- No crash-reporting service.
- No formal plugin marketplace or plugin signing.
- No macOS package yet.
